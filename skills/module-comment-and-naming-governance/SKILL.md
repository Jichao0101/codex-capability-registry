# 模块注释与命名一致性治理 Skill

## 1. Purpose

本 Skill 用于在模块级修改、重构、接口变更、状态模型调整时，系统性降低后续维护误解风险。

核心目标：

- 在模块级修改时补充必要的中文注释；
- 修复或指出命名与真实语义不一致的问题；
- 重点描述接口契约、状态模型、字段/成员语义、生命周期、不变量、副作用、异常/错误处理、线程安全、单位/坐标系/时间基准；
- 不追求注释数量，只追求降低误解风险；
- 不生成机械翻译代码或重复代码行为的注释。

本 Skill 的核心不是“多写注释”，而是让维护者更难误解代码。

## 2. When to use this Skill

### 强触发

遇到以下情况，应优先考虑调用本 Skill：

1. 用户明确要求补注释、统一注释风格、修复命名不一致；
2. 新增或重构 class、module、public API；
3. 修改跨模块接口、数据结构、状态机、缓存生命周期；
4. 修改关键 ownership、ID 分配、map key 语义等状态归属逻辑；
5. review 中指出代码难理解、注释不足、术语混乱；
6. 模块存在 legacy 命名，例如变量名与真实含义不一致；
7. 修改会影响调用者理解输入、输出、副作用或前置条件。

### 弱触发

遇到以下情况，可以谨慎调用本 Skill：

1. 修改核心函数内部逻辑；
2. 新增复杂局部变量、长生命周期变量、哨兵值变量；
3. 新增 fallback、缓存、异步、并发、状态迁移逻辑；
4. 修复 bug 时发现根因来自命名误导、注释过时或状态语义不清；
5. 新增阈值、单位敏感变量、坐标系敏感变量、时间戳变量。

弱触发时只允许处理本次触碰区域，不允许全文件泛化修改。

## 3. When NOT to use this Skill

以下情况禁止调用或默认不调用本 Skill。若用户没有明确授权修改代码，只能给出建议或说明不调用理由。

1. 用户只要求解释、评审、建议，没有要求修改代码；
2. 用户明确要求最小 diff、只修 bug、不要改注释或命名；
3. 单行修复、日志格式、空指针检查或纯格式化修改；
4. 热修复、回滚、临时绕过、紧急 patch；
5. 当前主任务是性能优化或分析，注释/重命名会干扰 review；
6. 语义证据不足，无法确认真实状态或字段含义；
7. 外部协议、配置 key、序列化字段、日志字段、ABI/API 约束变量不可改；
8. 改名会引发大范围调用链修改，但用户没有授权；
9. 已有注释虽然少，但变量作用域短、命名清晰、无跨模块误解风险；
10. 注释会把推测写成事实。

如果“不调用”与“可调用”条件同时出现，优先遵守不调用条件，除非用户明确要求继续。

## 4. Hard boundaries / Forbidden actions

触发本 Skill 后仍必须遵守以下硬边界：

1. 不得修改算法逻辑、状态迁移、阈值或调度策略；
2. 不得修改 public API 名称、参数、返回值，除非用户明确授权；
3. 不得修改外部协议字段、配置、序列化字段、日志字段；
4. 不得把不确定设计意图写成确定注释；
5. 不得生成机械复述代码注释，例如“遍历数组”“判断是否为空”“i 加一”；
6. 不得为了补注释而移动代码块、拆函数、重构控制流；
7. 不得大规模格式化无关代码；
8. 不得删除有价值注释，除非明显与代码不符；
9. 注释统一中文，不得引入英文注释；
10. 未运行测试时，不得声称行为完全不变或已验证，只能说明“本次目标是不改变行为，尚未验证”。

## 5. What to inspect

触发后检查以下对象，优先级从跨模块理解风险高的对象开始：

1. 文件级/模块级抽象；
2. class/struct 的职责边界；
3. public API；
4. private 核心函数；
5. 成员变量及容器 key/value 语义；
6. 长生命周期局部变量、哨兵值、fallback 逻辑；
7. ID、owner、session、token 等身份语义；
8. 状态机、缓存、队列、异步任务、锁、引用生命周期；
9. 阈值、单位、坐标系、时间基准；
10. 特殊返回值、nullptr、-1、NaN/Inf；
11. 副作用和错误处理策略。

## 6. Comment policy

### 6.1 类级

类级注释应说明：

- 该类代表的抽象语义；
- 是否维护状态；
- 状态如何随调用演化；
- ownership 或 identity 归属；
- 容器成员的 key/value 语义；
- 生命周期；
- 线程安全性；
- 与其他模块的边界。

### 6.2 接口级

public API 或跨模块函数优先补接口注释。注释应覆盖：

- 高层语义；
- 参数含义；
- 返回值；
- 输入输出关系；
- 前置条件；
- 后置条件；
- 副作用；
- 异常、错误、空值行为；
- 是否修改入参；
- 线程安全性；
- 单位、坐标系、时间基准。

允许使用 Doxygen 风格，但不要为了格式牺牲信息密度。

### 6.3 成员变量

成员变量注释应说明：

- 字段语义；
- 生命周期；
- key/value 含义；
- 默认值含义；
- 合法范围；
- 不变量；
- 与其他成员的耦合关系；
- 单位、坐标系、时间基准。

### 6.4 实现级

实现级注释只解释：

- 为什么这样做；
- 当前代码块的意图；
- 非显然技巧；
- 性能、数值、并发、兼容性约束；
- 为什么不用更简单方案；
- 关键风险和边界条件。

不得机械翻译代码。

### 6.5 局部变量

局部变量默认不加注释，除非满足至少一个条件：

- 跨越范围很大；
- 使用哨兵值；
- 维护状态不变量；
- 与其他变量必须同步更新；
- 单位、坐标系、时间基准不明显；
- 变量名无法表达生命周期语义。

## 7. Naming policy

检查并修复命名不一致，但必须克制。

优先处理：

1. 变量名与真实语义冲突；
2. legacy 命名已经误导维护者；
3. 同一概念出现多个术语；
4. map key 语义和变量名不一致；
5. owner、id、track、identity 等概念混用。

命名修复原则：

- 命名应表达真实 ownership；
- key 语义优先写入变量名；
- 同一模块内同一概念使用同一术语；
- 如果不能安全改名，保留原名并补充中文注释说明 legacy 语义；
- 改名必须同步所有引用，不能留下半改状态。

禁止：

- 未授权修改 public API 名称；
- 修改外部配置、序列化、日志协议字段；
- 为了“更好看”做主观改名；
- 大范围重命名但不更新所有引用；
- 命名和注释同时改变过多，导致 review 无法确认行为。

## 8. Workflow

触发后按以下步骤执行：

1. 判断是否真的应该调用本 Skill；
2. 如果属于禁止调用场景，只给出不调用理由或仅提供建议；
3. 识别本次修改范围；
4. 分析模块抽象、状态模型、接口契约、命名问题；
5. 区分确定事实与推测；
6. 先修复高风险命名不一致；
7. 再补充类级、接口级、成员级、核心私有函数级注释；
8. 清理机械复述型注释；
9. 保持 diff 最小；
10. 运行或建议运行可用验证；
11. 输出修改摘要、风险点、验证结果。

## 9. Output requirements

完成任务后必须输出：

- 修改摘要；
- 命名修复列表；
- 注释增强列表；
- 明确说明未修改算法逻辑；
- 未运行测试时说明未验证；
- 运行测试时提供命令和结果；
- 标出不确定语义或需要人工确认的地方。

## 10. Verification checklist

完成前自检：

- public API 是否未被误改；
- 外部协议、配置、序列化、日志字段是否未被误改；
- 是否未引入英文注释；
- 是否没有机械复述注释；
- 是否没有把推测写成事实；
- diff 是否没有扩大到无关区域；
- 所有重命名引用是否同步更新；
- 注释是否覆盖副作用和前置条件；
- 原算法行为是否保持不变；
- 未验证内容是否明确说明。

## 11. Style requirements

- 注释统一中文；
- 术语稳定，不要在同一模块混用多个词；
- 简洁，但必须表达契约和不变量；
- 优先解释机制、状态演化和约束；
- 不要写空泛注释；
- 不要写显而易见的注释；
- 对不确定语义保持克制。
