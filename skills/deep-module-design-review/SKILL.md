# 深模块设计评审

## 1. 定位

以“模块是否有效隐藏复杂性”为核心，而不是按行数、函数数或类大小机械拆分代码。

默认产物是设计评审与重构规划。除非用户明确授权实现，否则不要修改业务代码。

## 2. 核心判断

评审时始终区分两类边界：

- **对外抽象边界**：调用方看到的 public API、调用顺序、数据契约和失败模型。
- **内部实现组织**：private helper、pure helper、内部类型、策略对象、协作对象和文件布局。

先回答：

1. public API 是否已经足够窄？
2. 模块隐藏了多少复杂性，又把多少复杂性转移给调用方？
3. 调用方是否必须了解不必要的内部概念、步骤或组合规则？
4. 当前问题属于外部抽象浅，还是内部组织混乱？

如果 public API 已经窄且稳定，优先保留深接口，在内部整理实现。新增 public 或半 public 类型并非默认错误：当它形成更清晰、更深的独立抽象时，可以作为合理方案。

## 3. 深浅模块判据

### 深模块倾向

- public API 少且表达高层意图。
- 调用方不需要编排内部阶段。
- 解析、策略、状态、错误恢复或组合复杂性主要留在模块内部。
- API 的学习成本明显小于模块提供的功能复杂度。

### 浅模块倾向

- public API 暴露大量内部步骤、临时状态或执行顺序。
- 调用方需要跨多个对象拼装本应由模块隐藏的流程或规则。
- 新增类型只包装少量逻辑，却增加协调成本。
- 拆分后复杂性没有消失，只是转移到调用方。

不要仅因类大或函数长就判定模块浅。代码体积是信号，不是结论。

## 4. 先选择复杂性镜头

不要默认用状态密集 pipeline 的术语分析所有模块。先判断主要复杂性属于哪些类型：

- **业务编排**：用例、规则组合、事务边界、跨服务协作。
- **数据转换**：解析、校验、归一化、映射、计算、格式转换。
- **外部集成**：协议、认证、重试、超时、序列化、错误映射。
- **策略变化**：重试、分组、淘汰、排序、选择或定价规则。
- **资源管理**：连接、缓存、文件、内存、并发和清理。
- **状态与历史**：生命周期、撤销重做、会话、工作流或算法轨迹。
- **接口与兼容**：调用方契约、扩展点、版本和迁移成本。

只检查与该模块有关的维度：

- 对纯函数计算模块，重点分析输入模型、阶段组合、数值或算法边界，不强行寻找 state owner。
- 对配置解析和工具库，重点分析格式知识、校验、默认值、错误模型和可组合性。
- 对 HTTP client，重点分析传输细节、重试策略、认证、错误映射和调用方体验。
- 对缓存，重点分析存储职责与淘汰策略是否应分离。
- 对文本编辑器，重点分析文档状态、操作模型、历史记录和分组策略。
- 对状态密集 pipeline，才深入 owner、生命周期、不变量、状态流和副作用。

## 5. 评审流程

### 第一步：确认范围与行为契约

- 读取 public header、核心实现和直接相关类型。
- 确认允许修改范围、是否只做分析、是否允许改变 API 或行为。
- 记录必须保持的 API、输出、错误模型、兼容语义，以及任务相关的顺序或状态约束。
- 不从未读取的代码推断事实。

### 第二步：判断模块边界

- 描述模块提供的高层能力。
- 列出 public API 和调用方责任。
- 判定当前边界为深、浅或混合，并给出证据。
- 单独说明是否需要改变 public API。

### 第三步：识别复杂性类型与核心约束

- 选择适用的复杂性镜头，不要求每类模块都有状态模型。
- 找出领域规则、数据契约、策略变化点、外部依赖或长期状态。
- 识别模块必须独立维护的约束和调用方不应承担的知识。
- 仅在确实存在长期状态时，分析 owner、生命周期、状态流和不变量。

### 第四步：定位复杂性来源

按来源分类，不要只报告函数长度：

- 领域规则和业务编排。
- 数据转换、算法计算和校验。
- 外部协议、I/O 与错误处理。
- 策略选择和变化轴。
- 状态、历史或资源生命周期（若存在）。
- 兼容、扩展、日志和可观测性。

区分必要的领域复杂性和可由结构降低的阅读复杂性。

### 第五步：划分主流程与内部机制

主流程应保留：

- 能表达领域阶段和策略选择的编排。
- 体现高层用例、转换管线或协议交互的关键步骤。
- 对模块整体契约负责的决策。

优先提取：

- 无状态 pure helper。
- 副作用明确的 private helper。
- 表达领域输入、结果、错误或中间语义的小结构体。
- 可独立变化的内部策略函数或策略对象。
- 可隐藏协议、解析、验证、转换或存储细节的协作对象。

不要把一个大函数机械拆成大量只能按固定顺序调用、参数列表很长且共享隐式状态的小函数。

### 第六步：评估候选拆分

新增 public、半 public 或内部类时，不预设“少类更好”。检查：

1. 是否拥有稳定、可命名的职责？
2. 是否隐藏一组有内聚性的规则、策略、协议或状态？
3. 是否有自己的契约，并能独立测试或替换？
4. 是否让调用方使用更高层概念，而不是承担更多拼装步骤？
5. 可见性是否与真实复用范围相符：private、internal、package、public？
6. 是否避免循环依赖、职责重叠和含糊的协调所有权？

满足这些条件时，新增类型可能让系统形成多个更深的模块。例如：

- `TextBuffer`、`History`、`Operation`、`GroupingPolicy`
- `HttpClient`、`RetryPolicy`、`Transport`、`ErrorMapper`
- `Cache`、`Storage`、`EvictionPolicy`
- `ConfigParser`、`SchemaValidator`、`ResolvedConfig`

这些类型不要求都公开。根据调用方是否需要选择、替换、持有或组合它们，决定 public 或半 public 可见性。

不满足独立抽象条件时，再优先使用 private helper、小结构体或内部实现文件。

### 第七步：风险分级

每项建议必须标注：

- **低风险结构整理**：等价提取、命名局部结果、移动纯计算、收窄可见性并保持调用语义。
- **中风险边界重组**：引入协作类或策略对象、移动职责、调整错误或副作用边界，需要契约测试保护。
- **高风险行为或契约变更**：改变业务规则、算法、持久化格式、并发语义、失败模型、public API 或兼容行为。

不得把中高风险改动包装成“仅可读性优化”。

### 第八步：制定分阶段计划

通常按以下顺序：

1. 固化外部契约和关键行为基线。
2. 识别复杂性类型与稳定变化轴。
3. 提取 pure helper、值对象或局部语义结构。
4. 瘦身主流程或建立有内聚性的协作对象。
5. 在契约测试保护下调整内部边界。
6. 仅在收益明确时改变 public API 或迁移调用方。

每一阶段都写明验证方法和停止条件。

## 6. 不适用与停止条件

以下情况不要套用本 skill 做扩大分析：

- 用户要求明确、局部、机械的代码修改。
- 缺陷根因已确定且重构会扩大修复范围。
- 任务只涉及格式化、拼写、配置值或生成文件。
- 没有权限读取核心实现与接口，无法判断复杂性来源和契约。
- 用户要求直接实现，且已有经过审核的重构方案。

证据不足时，输出缺口和需要读取的最小范围，不要猜测模块深浅或状态语义。

## 7. 输出要求

使用 [templates/review-output-template.md](templates/review-output-template.md) 组织结果，至少包含：

1. 模块边界判断
2. public API 深浅分析
3. 主导可读性问题
4. 复杂性来源
5. 核心约束；状态密集模块再补 owner、生命周期与不变量
6. 应保留在主流程中的逻辑
7. 适合提取的内部机制
8. 拆分建议与暂缓项
9. 低/中/高风险改动分类
10. 分阶段重构计划
11. 验证策略

结论必须区分：

- 已由代码证实的事实
- 基于事实形成的设计判断
- 尚需验证的假设

## 9. 自检

结束前确认：

- 是否先分析 public API，再讨论拆类或拆函数？
- 是否区分外部边界和内部组织？
- 是否先选择了适合该模块的复杂性镜头？
- 是否只在必要时使用状态 owner、生命周期等概念？
- 是否检查复杂性是否被转移给调用方？
- 是否保留可理解的高层结构，而非把主流程或计算链拆碎？
- 是否公允评估了新增 public、半 public 或内部类的收益？
- 是否逐项标注行为风险？
- 是否给出可逐步验证、可停止的计划？
- 未获实现授权时，是否保持只读评审？
