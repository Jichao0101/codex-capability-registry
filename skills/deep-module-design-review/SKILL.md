# 深模块设计评审

## 1. 定位

以“模块是否隐藏复杂性”为核心，而不是按行数、函数数或类大小机械拆分代码。

默认只输出设计评审和重构规划。除非用户明确授权实现，否则不修改业务代码。

## 2. 基本评审顺序

1. 读取 public header、核心实现和直接相关类型或调用点。
2. 先判断 public API 是否足够窄，再讨论内部拆分。
3. 选择主要复杂性镜头，只分析相关维度。
4. 审计核心 orchestrator / phase 主函数是否具备高层叙述性。
5. 逐项审计已有 helper、private surface 和候选新增抽象。
6. 区分低风险结构整理、中风险边界重组、高风险行为或契约变更。
7. 给出分阶段计划、验证方法和停止条件。

证据不足时，说明缺口和最小补读范围，不猜测模块语义。

## 3. 复杂性镜头

按模块实际情况选择，不要强行套状态密集 pipeline 术语。

- 业务编排：用例、规则组合、事务边界、跨服务协作。
- 数据转换：解析、校验、归一化、映射、数值计算、格式转换。
- 外部集成：协议、认证、重试、超时、序列化、错误映射。
- 策略变化：重试、分组、淘汰、排序、选择、定价等变化轴。
- 资源管理：连接、缓存、文件、内存、并发和清理。
- 状态与历史：生命周期、撤销重做、会话、工作流或算法轨迹。
- 接口与兼容：调用方契约、扩展点、版本和迁移成本。

只有确实存在长期状态时，才分析 owner、生命周期、状态流和不变量。

## 4. 模块边界判断

先回答：

1. public API 是否少且表达高层意图？
2. 调用方是否需要编排内部步骤、临时状态或组合规则？
3. 模块隐藏了多少解析、策略、状态、错误恢复或组合复杂性？
4. 问题主要在外部抽象浅、内部主流程不可叙述，还是低层机制混杂？

结论可为：

- **深模块**：API 学习成本明显小于模块内部功能复杂度；调用方不承担内部编排。
- **浅模块**：API 暴露内部步骤、临时状态或执行顺序；复杂性转移给调用方。
- **混合**：public API 健康，但 private/header/internal surface 或主流程叙述性存在问题。

如果 public API 已经窄且稳定，优先保留外部接口，在内部整理实现。

## 5. 主流程叙述性审计

核心 orchestrator 或 phase 主函数应像高层状态机、转换管线或协议摘要，而不是把低层机制堆在同一层。

主流程应保留：

- 表达领域阶段、策略选择或协议交互的关键步骤。
- 对模块整体契约负责的决策。
- 状态推进、发布、提交、回滚、fallback 等高层动作。

主流程叙述性失败模式：

- helper 名称是执行步骤、容器操作或算法机制，而不是领域动作或 phase contract。
- 主流程层需要追踪临时集合、索引映射、matched/used 集合、fallback 标志、publish eligibility。
- phase 调用点看不出会修改哪些长期状态、输出、错误状态或生命周期计数。
- helper 只能按固定顺序调用；单独读任一 helper 不知道前置和后置条件。
- 高层 phase 之间夹杂 sanitize、loss matrix、row/column assignment、局部过滤、日志拼装等低层机制。
- 为缩短主函数而把小块提升为 private method，使 private/header surface 变成执行脚本。
- 为避免 private method 而保留复杂 lambda、闭包链或巨型 inline block，隐藏真实依赖和副作用。

## 6. Helper 与 Private Surface 审计

private section / package-internal API 也是认知接口。它不影响外部 ABI，并不意味着可以承载临时 helper、单帧中间类型或完整执行脚本。

新增 helper 前，必须先收敛已有 helper：

1. 删除无独立职责的浅包装。
2. 将纯算法、局部过滤、矩阵构造、索引映射、sanitize scratch 等降级到 `.cpp` internal、函数局部 block 或局部 lambda。
3. 合并只能固定顺序调用、共享同一批参数和隐式状态的 step helper。
4. 只有主流程仍不可叙述，且候选 helper 通过审计时，才新增 private phase helper。

逐项审计每个 private/helper：

- 名称是否表达完整领域动作、phase contract 或长期状态推进？
- 是否主要操作类长期状态、跨帧生命周期、资源状态或稳定配置？
- 参数是否只是单帧事实、低层算法中间态或 scratch state？
- 副作用是否集中且显式，调用点能否看出它推进、清理、发布、提交或回滚什么？
- 是否有独立契约、复用、测试或失败边界？
- 是否只能按固定顺序与相邻 helper 工作；若是，优先合并为更深 phase-level helper。
- 替代方案是否更差：`.cpp` helper 是否会参数膨胀，局部 lambda 是否会隐藏依赖，inline block 是否破坏主流程叙述？

private/header surface 优先保留：

- 长期状态字段。
- public API 依赖的稳定类型。
- 真正跨 phase 的契约类型。
- 少量表达完整语义动作的 phase-level private methods。
- 对主流程叙述性有明显收益、主要操作长期状态、副作用集中，且替代方案更差的稳定子流程 helper。

private/header surface 默认不放：

- 单帧临时 row、view、payload、context。
- solver row 解释器、索引映射器、eligibility 搬运结构。
- 只在一个函数链内使用的中间结果。
- 只能按固定顺序调用的机械步骤 helper 全集。

### C++ 完整类型与实现隐藏

审计 C++ header 中的 private 类型时，先区分“private 访问控制”和“定义对调用方可见”。不得仅因某类型属于实现细节，就建议把它的定义迁到 `.cpp`。

- 若类按值持有该类型，成员声明处通常要求完整类型；将类型定义直接移到 `.cpp` 会使对象布局无法确定，方案本身不成立。
- 若该类型只用于 private 方法声明的参数或返回值，或以指针/引用成员存在，不参与 owning class 的对象布局，才可能通过前置声明和 `.cpp` 定义收窄可见性；仍需按具体声明验证不完整类型是否合法。
- 若按值成员必须真正隐藏，通常需要改为指针间接层，例如 PImpl 或 internal impl class，并审计 ownership、析构点、拷贝/移动语义、异常保证、分配成本、ABI 和构建依赖。
- PImpl/internal impl class 会改变表示与职责边界。即使 public API 文本不变，也至少属于中风险边界重组；不得标为低风险结构整理或“只移动实现细节”。
- 若不值得承担间接层成本，允许 private nested type 保留在 header。评审应说明它泄露的是定义和构建依赖，不等于调用方获得了可访问的 public API。

给出迁移建议前必须回答：

```text
当前使用方式：按值成员 / 指针成员 / 引用成员 / 仅用于方法签名
header 是否要求完整类型：
仅前置声明是否可编译：
真正隐藏是否需要 PImpl/internal impl：
ownership 与 special members 如何变化：
风险级别：
```

## 7. 参数过多时的处理

不得直接通过成员化或 `Context` / `Payload` 打包解决参数过多。先区分参数来源：

- **类长期状态**：对象身份、跨帧状态、缓存、资源、配置、生命周期状态。可由 private method 直接访问；若反复显式传递，应审计是否本来就是类职责。
- **单帧事实**：本帧输入、候选集合、当前领域事实、输出快照。优先保持局部变量或显式参数，不要为了少传参而成员化。
- **低层算法中间态**：solver row、index map、loss matrix、temporary set、eligibility mask、scratch。优先限制在 `.cpp` helper 或局部 block。
- **共享参数组**：多个 helper 反复共享同一批参数时，优先判断是否应合并为更深 phase-level helper。

只有参数本身具有跨 phase、跨帧、ownership、生命周期、不变量或失败语义时，才考虑成员化或稳定类型化。

## 8. 新增抽象必要性审计

新增 `struct`、`class`、`enum`、`View`、`Payload`、`Context`、`Row`、`Result`、`Helper` 不是默认优化。只有它能隐藏复杂性、承载不变量、降低组合错误或形成稳定职责时才成立。

优先级默认从低可见性开始：

```text
已有结构 + 清晰变量名
  > type alias
  > 局部变量 / 局部 lambda
  > 函数局部 struct
  > cpp/internal struct
  > private nested struct
  > internal class
  > public class
```

新增抽象前检查：

- 是否已有 domain object、map、set、vector 或其他结构可以表达同一契约？
- 是否只是已有字段的重排、筛选、浅包装或搬运？
- 是否引入真实的新生命周期、所有权、不变量或失败语义？
- 是否降低调用方或上层 phase 必须理解的组合规则？
- 是否有独立测试、替换、复用或演进价值？

对每个建议新增的类型，必须回答：

```text
新增类型：
解决的问题：
新增的不变量：
为什么不能复用已有类型：
如果不用该类型，替代方案是什么：
收益是否大于新增阅读入口：
建议可见性：
```

局部、只为缩短复杂类型名、且不跨函数传播的 type alias 可简化说明。

已有低收益抽象也要审计。若不通过，优先删除、局部化、降级为 type alias、移入 `.cpp` 或合并回所属 phase-level helper；不要用长注释、新 wrapper 或解释层补偿。

其中“移入 `.cpp`”只适用于不要求 owning class 在 header 中看到完整定义的类型或机制。按值 private 成员不得套用该默认处置；若需隐藏，按 C++ 完整类型与实现隐藏规则单独评估。

## 9. 拆分与风险分级

不要把大函数机械拆成大量只能固定顺序调用、参数高度重叠、共享隐式状态的小函数。优先保留完整领域动作或 phase contract。

风险必须标注：

- **低风险结构整理**：等价提取、命名局部结果、移动纯计算、收窄可见性且保持调用语义。
- **中风险边界重组**：引入协作类或策略对象、用 PImpl/internal impl 隐藏原本按值持有的 private 类型、移动职责、调整错误或副作用边界，需要契约测试保护。
- **高风险行为或契约变更**：改变业务规则、算法、持久化格式、并发语义、失败模型、public API 或兼容行为。

不得把中高风险改动包装成“仅可读性优化”。

## 10. 分阶段计划

通常按以下顺序：

1. 固化外部契约和关键行为基线。
2. 识别复杂性类型、长期状态和稳定变化轴。
3. 先收敛已有 helper 和 private/header surface。
4. 提取或局部化 pure helper、内部策略、低层机制。
5. 瘦身主流程，必要时建立少量 phase-level helper 或协作对象。
6. 在契约测试保护下调整内部边界。
7. 仅在收益明确时改变 public API 或迁移调用方。

每一阶段都写明验证方法和停止条件。

## 11. 输出要求

正式评审使用 [templates/review-output-template.md](templates/review-output-template.md)，只保留与任务有关的内容。

至少覆盖：

1. 评审范围和必须保持的行为。
2. 模块边界判断和 public API 深浅分析。
3. 内部主流程叙述性评估。
4. Private/helper surface 逐项审计。
5. 参数来源与 Context/Payload/成员化风险。
6. 主导可读性问题和复杂性来源。
7. 核心约束；状态密集模块补 owner、生命周期、不变量。
8. 应保留在主流程中的逻辑。
9. 适合提取、降级、合并或删除的内部机制。
10. 新增抽象必要性审计和实现前类型约束清单。
11. 低/中/高风险分类。
12. 分阶段重构计划和验证策略。
13. 已证实事实、设计判断、待验证假设和未解决问题。

## 12. 停止条件

以下情况不要扩大分析：

- 用户要求明确、局部、机械的代码修改。
- 缺陷根因已确定且重构会扩大修复范围。
- 任务只涉及格式化、拼写、配置值或生成文件。
- 没有权限读取核心实现与接口，无法判断复杂性来源和契约。
- 用户要求直接实现，且已有经过审核的重构方案。

## 13. 自检

结束前确认：

- 是否先分析 public API，再讨论拆类或拆函数？
- 是否区分外部边界、内部叙述边界和低层实现机制？
- 是否在 public API 健康时仍审计了内部主流程？
- 是否选择了适合该模块的复杂性镜头？
- 是否检查复杂性是否被转移给调用方？
- 是否逐项审计 private/helper，而不是泛泛说“可以提 helper”？
- 是否在新增 helper 前先收敛已有 helper？
- 参数过多时，是否先区分长期状态、单帧事实、低层中间态和共享参数组？
- 是否避免直接用成员化、`Context` 或 `Payload` 掩盖参数来源？
- 是否防止 header/private surface 变成内部执行脚本？
- 对 C++ 按值 private 类型，是否检查了完整类型和对象布局要求，而不是仅因“实现细节”建议迁到 `.cpp`？
- 若建议 PImpl/internal impl，是否审计 ownership、special members、ABI/构建影响，并将其标为边界重组？
- 是否对新增类型做反事实比较，并审计已有低收益抽象？
- 是否区分低/中/高风险，并给出可验证、可停止的计划？
- 未获实现授权时，是否保持只读评审？
