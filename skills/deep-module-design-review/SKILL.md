# 深模块设计评审

## 1. 定位

以“模块是否有效隐藏复杂性”为核心，而不是按行数、函数数或类大小机械拆分代码。

默认产物是设计评审与重构规划。除非用户明确授权实现，否则不要修改业务代码。

## 2. 核心判断

评审时始终区分两类边界：

- **对外抽象边界**：调用方看到的 public API、调用顺序、数据契约和失败模型。
- **内部实现组织**：private helper、pure helper、内部类型、策略对象、协作对象和文件布局。

先回答：

1. public API 是否已经足够窄？
2. 模块隐藏了多少复杂性，又把多少复杂性转移给调用方？
3. 调用方是否必须了解不必要的内部概念、步骤或组合规则？
4. 当前问题属于外部抽象浅，还是内部组织混乱？

如果 public API 已经窄且稳定，优先保留深接口，在内部整理实现。新增 public 或半 public 类型并非默认错误：当它形成更清晰、更深的独立抽象时，可以作为合理方案。

## 3. 深浅模块判据

### 深模块倾向

- public API 少且表达高层意图。
- 调用方不需要编排内部阶段。
- 解析、策略、状态、错误恢复或组合复杂性主要留在模块内部。
- API 的学习成本明显小于模块提供的功能复杂度。

### 浅模块倾向

- public API 暴露大量内部步骤、临时状态或执行顺序。
- 调用方需要跨多个对象拼装本应由模块隐藏的流程或规则。
- 新增类型只包装少量逻辑，却增加协调成本。
- 拆分后复杂性没有消失，只是转移到调用方。

不要仅因类大或函数长就判定模块浅。代码体积是信号，不是结论。

## 4. 先选择复杂性镜头

不要默认用状态密集 pipeline 的术语分析所有模块。先判断主要复杂性属于哪些类型：

- **业务编排**：用例、规则组合、事务边界、跨服务协作。
- **数据转换**：解析、校验、归一化、映射、计算、格式转换。
- **外部集成**：协议、认证、重试、超时、序列化、错误映射。
- **策略变化**：重试、分组、淘汰、排序、选择或定价规则。
- **资源管理**：连接、缓存、文件、内存、并发和清理。
- **状态与历史**：生命周期、撤销重做、会话、工作流或算法轨迹。
- **接口与兼容**：调用方契约、扩展点、版本和迁移成本。

只检查与该模块有关的维度：

- 对纯函数计算模块，重点分析输入模型、阶段组合、数值或算法边界，不强行寻找 state owner。
- 对配置解析和工具库，重点分析格式知识、校验、默认值、错误模型和可组合性。
- 对 HTTP client，重点分析传输细节、重试策略、认证、错误映射和调用方体验。
- 对缓存，重点分析存储职责与淘汰策略是否应分离。
- 对文本编辑器，重点分析文档状态、操作模型、历史记录和分组策略。
- 对状态密集 pipeline，才深入 owner、生命周期、不变量、状态流和副作用。

## 5. 评审流程

### 第一步：确认范围与行为契约

- 读取 public header、核心实现和直接相关类型。
- 确认允许修改范围、是否只做分析、是否允许改变 API 或行为。
- 记录必须保持的 API、输出、错误模型、兼容语义，以及任务相关的顺序或状态约束。
- 不从未读取的代码推断事实。

### 第二步：判断模块边界

- 描述模块提供的高层能力。
- 列出 public API 和调用方责任。
- 判定当前边界为深、浅或混合，并给出证据。
- 单独说明是否需要改变 public API。

### 第三步：识别复杂性类型与核心约束

- 选择适用的复杂性镜头，不要求每类模块都有状态模型。
- 找出领域规则、数据契约、策略变化点、外部依赖或长期状态。
- 识别模块必须独立维护的约束和调用方不应承担的知识。
- 仅在确实存在长期状态时，分析 owner、生命周期、状态流和不变量。

### 第四步：定位复杂性来源

按来源分类，不要只报告函数长度：

- 领域规则和业务编排。
- 数据转换、算法计算和校验。
- 外部协议、I/O 与错误处理。
- 策略选择和变化轴。
- 状态、历史或资源生命周期（若存在）。
- 兼容、扩展、日志和可观测性。

区分必要的领域复杂性和可由结构降低的阅读复杂性。

### 第五步：划分主流程与内部机制

主流程应保留：

- 能表达领域阶段和策略选择的编排。
- 体现高层用例、转换管线或协议交互的关键步骤。
- 对模块整体契约负责的决策。

优先提取：

- 无状态 pure helper。
- 副作用明确的 private helper。
- 表达稳定领域输入、结果或错误契约的小结构体。
- 可独立变化的内部策略函数或策略对象。
- 可隐藏协议、解析、验证、转换或存储细节的协作对象。

上述内容只是候选，不构成默认提取理由。任何新 helper 或小结构体仍必须通过第六步的抽象必要性审计。

表达中间语义的小结构体必须特别审慎。默认先使用局部变量、type alias、局部 lambda 或函数局部 struct；不能仅因它给中间步骤命名，就直接提升为 private header、跨 phase 契约或稳定模块概念。

不要把一个大函数机械拆成大量只能按固定顺序调用、参数列表很长且共享隐式状态的小函数。

#### 层级接口优先于步骤接口

- 优先保留表达完整领域动作或 phase contract 的函数，而不是把内部执行脚本展开成大量 step helper。
- 优先使用 domain-level fact，而不是让 solver index、临时 row、输出 eligibility 等机制事实跨层传播。
- 优先复用已有表示，而不是为每个步骤新增 payload、view 或 result wrapper。
- 如果多个 helper 只能按固定顺序共同完成一个语义动作、参数高度重叠且共享同一隐式状态，应优先收敛为一个更深的 phase-level helper。
- 只有步骤具有独立契约、独立复用、独立测试或独立失败边界时，才建议暴露为单独 private method。

例如，默认优先考虑：

```text
updateBodyStateFromDetections(...)
finalizeBodyStateForFrame(...)
publishBodyTracks(...)
```

不要默认展开为：

```text
solveBodyAssignments(...)
applyBodyMatches(...)
advanceBodyMiss(...)
finalizeBodyTracksForOutput(...)
projectBodyTracks(...)
publishBodyTracks(...)
```

后者只有在各步骤确实拥有独立契约、复用或测试价值时才成立。

### 第六步：评估候选拆分

新增 public、半 public 或内部类时，不预设“少类更好”。检查：

1. 是否拥有稳定、可命名的职责？
2. 是否隐藏一组有内聚性的规则、策略、协议或状态？
3. 是否有自己的契约，并能独立测试或替换？
4. 是否让调用方使用更高层概念，而不是承担更多拼装步骤？
5. 可见性是否与真实复用范围相符：private、internal、package、public？
6. 是否避免循环依赖、职责重叠和含糊的协调所有权？

满足这些条件时，新增类型可能让系统形成多个更深的模块。例如：

- `TextBuffer`、`History`、`Operation`、`GroupingPolicy`
- `HttpClient`、`RetryPolicy`、`Transport`、`ErrorMapper`
- `Cache`、`Storage`、`EvictionPolicy`
- `ConfigParser`、`SchemaValidator`、`ResolvedConfig`

这些类型不要求都公开。根据调用方是否需要选择、替换、持有或组合它们，决定 public 或半 public 可见性。

不满足独立抽象条件时，再优先使用 private helper、小结构体或内部实现文件。

#### 抽象必要性审计

新增 `struct`、`class`、`View`、`Payload`、`Context`、`Result` 或 `Helper` 不是默认优化。只有当新抽象能隐藏复杂性、承载不变量、降低调用方拼装成本或形成稳定职责时才成立；不能仅因为“语义更清楚”就新增类型。

新增抽象前，按以下顺序检查是否已有更轻的表示足够：

```text
已有结构 + 清晰变量名
  > type alias
  > 局部变量 / 局部 lambda
  > 函数局部 struct
  > cpp/internal struct
  > private nested struct
  > internal class
  > public class
```

必须逐项检查：

- 是否已有 domain object、map、set、vector 或其他结构可以表达同一契约？
- 是否可通过 `const` 容器、清晰变量名、type alias、局部变量或局部 lambda 表达？
- 新类型是否只是已有字段的重排、筛选、浅包装或搬运？
- 新类型是否引入了真实的新生命周期、所有权、不变量或失败语义？
- 新类型是否增加了新的阅读入口、构造规则、转换步骤和跨层传播？

新增抽象至少应满足以下一项正当性条件：

- 承载新的不变量，并能由类型或构造入口维护。
- 隐藏一组稳定且内聚的复杂规则、策略、协议或状态。
- 具有独立生命周期或所有权语义。
- 被多个调用点稳定复用，并实质减少重复逻辑或组合错误。
- 降低调用方必须理解的组合规则，而不是转移复杂性。
- 形成可独立测试、替换或演进的深模块边界。

如果结构只把前一步结果搬到后一步，或只给局部事实换名，不应默认新增结构体。

审计不仅针对拟新增抽象，也必须覆盖已有抽象。已有类型、helper 或 wrapper 若不能通过正当性测试，不要把“已经存在”视为保留理由。

发现已有低收益抽象时，优先按以下方向评估：

1. 删除，直接复用已有表示。
2. 降级为局部变量或局部 lambda。
3. 降级为 type alias。
4. 移入 `.cpp` anonymous namespace 或 internal 实现区。
5. 合并回唯一调用函数或所属 phase-level helper。

不要优先通过增加注释、补充 wrapper、再新增解释层或扩大命名体系来维持低收益抽象。若抽象的存在意义主要依赖长注释说明，而代码本身没有独立不变量、生命周期、规则或复用价值，应把它视为删除或降级信号。

以下抽象默认不建议引入，除非给出明确反证：

- 只包装已有 map、set、vector 或 domain object 的 `View`。
- 只在单个函数链中传递的 `Payload`、`Context` 或 `Row`。
- 同时混合 solver index、状态指针、领域对象和输出约束的中间结构。
- 只为让阶段看起来“清楚”而新增的 result wrapper。
- 只依靠注释解释存在意义、但代码没有独立职责的 helper。
- 把固定调用顺序固化到 header 或 private API surface 的步骤函数。

这不是禁止新增抽象。若候选类型通过上述正当性测试，并形成更深的职责边界，应明确建议保留或新增。

#### Private Header Surface 审计

对 C++ 或 header-style 模块，private section 也是阅读者必须理解的认知接口。它不影响外部 ABI，并不意味着可以无限承载临时 helper、单帧中间类型和完整执行脚本。

private header 中优先只保留：

- 长期状态字段。
- public API 依赖的稳定类型。
- 真正跨 phase 的契约类型。
- 少量表达完整语义动作的 phase-level private methods。

private header 中默认不应放入：

- 单帧临时 row、view、payload 或 context。
- solver row 解释器、索引映射器或 eligibility 搬运结构。
- 只在一个函数链内使用的中间结果。
- 只能按固定顺序调用的机械步骤 helper 全集。

如果 helper 或类型只服务一个 phase，优先放在 `.cpp` anonymous namespace、函数局部 struct、局部 lambda 或局部 block 中。只有当其跨 phase 契约稳定、需要多个成员函数共享，且收益大于新增认知入口时，才提升到 private header。

#### 反事实比较

每个新增抽象建议都必须同时回答：

```text
新增类型：
解决的问题：
新增的不变量：
为什么不能复用已有类型：
如果不用该类型，替代方案是什么：
收益是否大于新增阅读入口：
建议可见性：
```

如果无法具体回答，不应建议新增该抽象。若替代方案使用已有表示即可清晰完成任务，应优先选择替代方案。

仅用于缩短函数局部复杂类型名、且不跨函数边界传播、不引入新语义或不变量的 type alias，可使用简化审计：用一行说明原类型、alias 用途和局部作用域，不要求填写完整反事实表。若 alias 开始跨函数、跨文件或承担领域语义，则恢复完整审计。

#### 实现前新增类型约束与交接清单

本 skill 只输出实现前的设计约束和审计清单，不直接规定实现行为，不执行代码修改，也不替代后续实现任务说明。

若后续进入实现阶段，应把本评审中的抽象必要性审计作为实现前置检查。评审需要明确哪些候选类型已通过审计、哪些未通过，以及建议的最低必要可见性，供实现任务引用。

前置审计覆盖：

- `struct`、`class`、`enum`、`View`、`Payload`、`Context`、`Row`、`Result` 和 wrapper。
- private nested type、`.cpp` internal type、函数局部 named struct。
- 为现有类型重新包装语义的 type alias；简单缩短冗长类型名的 alias 也应说明必要性，但可使用简化审计。

通过审计至少要求：

1. 已完成反事实比较。
2. 已说明新增不变量、稳定职责、生命周期、复用或深边界收益。
3. 已证明已有表示、局部变量或局部 lambda 不足。
4. 已选择最低必要可见性。
5. 已在方案中明确记录审计结论，而不是留给实现阶段补写理由。

评审输出应形成“已通过审计的候选类型清单”和“未通过或暂缓的候选类型清单”。该清单是后续实现的设计输入，不是本 skill 对实现代理的运行控制。

清单详细程度应与任务规模和抽象风险匹配：

- 小型评审、没有新增抽象建议或用户只要求轻量判断时，可直接写“无新增类型建议 / 无需交接清单”，不机械展开完整表格。
- 只有一个低风险局部 type alias 时，可用一行简化记录。
- 涉及 private header、跨 phase 类型、生命周期、不变量、所有权或 public/internal API 时，使用完整审计清单。

若后续实现任务发现清单外的新类型需求，本评审建议重新执行或补充抽象必要性审计，并更新设计约束。具体如何暂停、变更或继续实现，由实现任务说明和项目治理决定。

评审结论不应接受“先新增类型，再通过注释、测试或既成代码反向证明合理性”作为设计依据。测试可以验证行为，但不能替代抽象必要性证明。

以下情况仍允许新增类型，但同样必须先通过审计：

- 外部协议、schema、ABI 或代码生成契约强制要求。
- 新类型承载无法由现有类型维护的安全性或正确性不变量。
- 新协作对象形成明确、更深且可独立演进的模块边界。

该前置约束的目标是让“复用已有表示”成为实现方案的默认设计，不是禁止形成真正有价值的深抽象。

### 第七步：风险分级

每项建议必须标注：

- **低风险结构整理**：等价提取、命名局部结果、移动纯计算、收窄可见性并保持调用语义。
- **中风险边界重组**：引入协作类或策略对象、移动职责、调整错误或副作用边界，需要契约测试保护。
- **高风险行为或契约变更**：改变业务规则、算法、持久化格式、并发语义、失败模型、public API 或兼容行为。

不得把中高风险改动包装成“仅可读性优化”。

### 第八步：制定分阶段计划

通常按以下顺序：

1. 固化外部契约和关键行为基线。
2. 识别复杂性类型与稳定变化轴。
3. 提取 pure helper、值对象或局部语义结构。
4. 瘦身主流程或建立有内聚性的协作对象。
5. 在契约测试保护下调整内部边界。
6. 仅在收益明确时改变 public API 或迁移调用方。

每一阶段都写明验证方法和停止条件。

## 6. 不适用与停止条件

以下情况不要套用本 skill 做扩大分析：

- 用户要求明确、局部、机械的代码修改。
- 缺陷根因已确定且重构会扩大修复范围。
- 任务只涉及格式化、拼写、配置值或生成文件。
- 没有权限读取核心实现与接口，无法判断复杂性来源和契约。
- 用户要求直接实现，且已有经过审核的重构方案。

证据不足时，输出缺口和需要读取的最小范围，不要猜测模块深浅或状态语义。

## 7. 输出要求

使用 [templates/review-output-template.md](templates/review-output-template.md) 组织结果，至少包含：

1. 模块边界判断
2. public API 深浅分析
3. 主导可读性问题
4. 复杂性来源
5. 核心约束；状态密集模块再补 owner、生命周期与不变量
6. 应保留在主流程中的逻辑
7. 适合提取的内部机制
8. 拆分建议与暂缓项
9. 低/中/高风险改动分类
10. 分阶段重构计划
11. 验证策略
12. 新增抽象必要性审计
13. 实现前新增类型约束与交接清单

结论必须区分：

- 已由代码证实的事实
- 基于事实形成的设计判断
- 尚需验证的假设

“新增抽象必要性审计”必须区分：

- 建议保留或新增的深抽象。
- 建议降级为函数局部或 `.cpp` 实现细节的抽象。
- 建议删除的浅包装。
- 不建议新增抽象、应复用已有表示的场景。

对每个建议新增的类型使用第六步的反事实比较格式。即使模板尚未包含该小节，也要在正式评审末尾追加。
对已有但不通过审计的抽象，必须明确给出删除、局部化、type alias、移入 `.cpp` 或合并回调用函数的处置建议；不得只建议补注释。
局部且不跨函数传播的纯缩写 type alias 可按简化审计记录，不要求完整反事实表。

若任务包含或后续可能进入实现，“实现前新增类型约束与交接清单”必须列出：

- 已通过审计的候选类型及其最低必要可见性。
- 明确拒绝或暂缓的候选类型。
- 当前方案能否在不新增类型的前提下实施。
- 后续出现清单外类型需求时，需要补充审计的设计建议。

该清单只作为后续实现任务的输入；不得在本 skill 中直接修改代码或替代实现任务的范围、步骤、验证和治理要求。
若无新增类型建议或任务规模较小，可明确写“无需交接清单”，不为满足格式机械展开空表。

## 9. 自检

结束前确认：

- 是否先分析 public API，再讨论拆类或拆函数？
- 是否区分外部边界和内部组织？
- 是否先选择了适合该模块的复杂性镜头？
- 是否只在必要时使用状态 owner、生命周期等概念？
- 是否检查复杂性是否被转移给调用方？
- 是否保留可理解的高层结构，而非把主流程或计算链拆碎？
- 是否公允评估了新增 public、半 public 或内部类的收益？
- 是否在新增类型前检查了已有表示是否足够？
- 是否把 private header surface 也当作认知接口审计？
- 是否避免把单帧临时结构、solver row、payload、context 暴露成稳定概念？
- 是否区分 phase-level 深接口和 step-level 机械 helper？
- 是否对每个新增抽象做了反事实比较？
- 是否对纯局部 type alias 使用了与风险匹配的简化审计，而非机械套完整表格？
- 若后续进入实现，是否形成了可供实现任务引用的已通过审计类型清单？
- 是否明确说明清单外类型需要补充设计审计，而没有越权规定实现代理的运行行为？
- 是否让交接清单详细度与任务规模和抽象风险匹配，并允许无新增类型时省略完整表格？
- 是否保持本 skill 只输出设计约束和审计清单，不直接修改代码或替代实现任务说明？
- 是否避免用注释补偿低收益抽象？
- 是否审计了已有抽象，而不只审计拟新增抽象？
- 对不通过审计的已有抽象，是否优先建议删除、降级、局部化或合并，而不是补注释或新增解释层？
- 是否逐项标注行为风险？
- 是否给出可逐步验证、可停止的计划？
- 未获实现授权时，是否保持只读评审？
